# DESIGN.md — romi-01 System Architecture

> **Purpose**: Machine-readable reference for AI agents, scripts, and automation.
> This file is the single entry point for understanding the entire system.
> If you are an AI reading this, start here, then follow references as needed.

## System Identity

- **Name**: romi-01
- **Type**: Personal growth tracking, learning, and self-coaching system
- **Owner**: Romi (see `context/about-me.md` for full profile)
- **Philosophy**: Stoic — discipline over motivation, action over intention
- **Primary interface**: VS Code + Claude extension, CLI scripts, future: SMS/messaging
- **Data format**: Markdown files with YAML frontmatter, JSON config
- **Version control**: Git

## Design Principles

1. **Config-driven**: All modular behavior reads from `config/`. Adding a category, changing a prompt, or adjusting tracking requires editing config — not rewriting scripts.
2. **Plain text first**: Everything is markdown or JSON. No databases, no proprietary formats. Human-readable, machine-parseable, git-friendly.
3. **Low friction**: The system must be usable in under 5 minutes per day. Any workflow that takes longer should be optional.
4. **Input-agnostic**: The system accepts input from multiple channels — manual file editing, CLI scripts, Claude conversation, and (future) SMS/text messages. All channels write to the same data format.
5. **Trend-native**: Every data point captured should contribute to long-term trend analysis. Frontmatter fields exist for machine parsing; prose sections exist for human reflection.
6. **Separation of concerns**: Config defines structure. Scripts handle I/O. Prompts define AI behavior. Data lives in dated files. Nothing is tangled.

### Daily Entry Format (canonical data structure)

Every daily entry follows this exact structure. All input channels must produce
output conforming to this format. All analysis tools must be able to parse it.

```markdown
---
date: YYYY-MM-DD
exercise: ~          # minutes — optional, null if not given
sleep: ~             # hours — optional, null if not given
mood: ~              # 1-5 rating — optional
reading: ~           # chapters/units — optional
work: ~              # 1-5 rating — optional
learning: false      # boolean
skills: false        # boolean
tags: []             # entry-level tags — set by AI via tag-update prompt or manually
---

# YYYY-MM-DD — DayName

## 🎯 Morning Intention
<!-- prompt from config -->

## 🧠 What I Learned
<!-- prompt from config -->

[...additional categories from config/categories.json...]
```

**Frontmatter rules**:
- Fields come from `trackFields` of composite categories and `id` of simple trackable categories
- Boolean fields default to `false`; numeric fields default to `~` (null/not given)
- `tags: []` is an entry-level tag list — AI-detected or manually added; distinct from category-level tags in config
- Frontmatter is the machine-parseable layer; prose sections are the human layer
- Both layers describe the same day but serve different consumers

**Category section rules**:
- Sections appear in the order defined in `config/categories.json`
- Format: `## {icon} {name}\n<!-- {prompt} -->\n\n{content}\n`
- Empty sections retain their headers and prompts (for manual fill-in)
- Content length: `minLines` to `maxLines` as defined in config

## Messaging / SMS Integration (Future — Phase 2)

### Vision
Romi can receive a text from the system (morning prompt, reminder, accountability nudge)
and reply via SMS to log entries, answer check-ins, or request information. Replies are
parsed and written to the daily entry file, same as any other input channel.

### Architecture (planned)

```
┌─────────────┐    ┌──────────────────┐    ┌─────────────┐
│ Romi's Phone │◄──►│ Messaging Gateway │◄──►│ romi-01 API │
│ (SMS/iMsg)   │    │ (Twilio / ntfy /  │    │ (Node.js    │
│              │    │  Pushover / etc)  │    │  service)   │
└─────────────┘    └──────────────────┘    └──────┬──────┘
                                                   │
                                           ┌───────▼───────┐
                                           │ Parse & Route  │
                                           │ - Identify     │
                                           │   category     │
                                           │ - Extract value│
                                           │ - Write to     │
                                           │   daily entry  │
                                           └───────┬───────┘
                                                   │
                                           ┌───────▼───────┐
                                           │ daily/YYYY-MM/ │
                                           │ YYYY-MM-DD.md  │
                                           └───────────────┘
```

### Message Types (outbound — system to Romi)

| Type | Trigger | Content |
|------|---------|---------|
| Morning prompt | Scheduled (e.g., 7:00 AM) | Stoic thought + micro-challenge (from `morning-start.md`) |
| Log reminder | Scheduled (e.g., 9:00 PM) | "What did you learn today?" or specific category prompt |
| Streak alert | Missed day detected | "You missed yesterday. One line is enough. What did you learn?" |
| Accountability nudge | Consistency check fails threshold | "3 days without exercise. 10 min walk today?" |
| Weekly prompt | Sunday morning | "Time for your weekly review. Ready?" |

### Message Types (inbound — Romi to system)

| Format | Parsed As | Example |
|--------|-----------|---------|
| `learning: <text>` | Category log entry | `learning: promises vs callbacks in depth` |
| `exercise: <number> <text>` | Trackable + prose | `exercise: 25 walked the neighborhood` |
| `mood: <1-5>` | Rating update | `mood: 4` |
| `sleep: <number>` | Numeric update | `sleep: 7.5` |
| Free text (no prefix) | Routed to reflection or auto-categorized by AI | `Rough day at work but kept my cool` |
| `skip` | Acknowledged skip, no data written | `skip` |
| `status` | Triggers status.js output sent back | `status` |

### Design Constraints for SMS Compatibility
1. **All input parsing must handle short, informal text.** No one types perfect markdown on a phone.
2. **Category IDs from `config/categories.json` serve as message prefixes.** The parser matches the first word against known category IDs.
3. **AI fallback**: If a message doesn't match a known prefix, pass it to Claude for classification using the category config as reference.
4. **Idempotent writes**: Logging the same category twice in one day appends, never overwrites.
5. **Confirmation**: After parsing and writing, send a brief confirmation back: "✓ Logged exercise: 25 min — walked the neighborhood"

### Implementation Options (evaluate when ready)

| Option | Pros | Cons | Cost |
|--------|------|------|------|
| **Twilio SMS** | Real SMS, reliable, two-way | Requires server, per-message cost | ~$1/mo + $0.0079/msg |
| **ntfy.sh** | Free, self-hostable, push notifications | Not true SMS, needs app | Free |
| **Pushover** | Simple API, reliable push | One-way only (no reply parsing) | $5 one-time |
| **iMessage via Shortcuts** | Native to iPhone, free | Apple-only, fragile automation | Free |
| **Telegram Bot** | Free, two-way, rich formatting | Requires Telegram app | Free |
| **Email-based** | Universal, free | Slower, less conversational | Free |

### Recommended Path
Start with **Telegram Bot** or **ntfy.sh** for push notifications (outbound).
Add **Twilio** or **Telegram reply parsing** for inbound when ready.
Keep the parser module separate so the messaging transport is swappable.

## AI Agent Instructions

If you are an AI agent (Claude, Claude Code, or a future automation) working within
this repo, follow these rules:

1. **Read `context/about-me.md` first** in any conversation involving coaching, review, or personal context.
2. **Read `config/categories.json`** before creating, parsing, or analyzing daily entries. Never hardcode category names.
3. **Read the relevant prompt file** from `prompts/` if one exists for the task. Follow its instructions exactly.
4. **Write daily entries in the canonical format** described above. Frontmatter + category sections, in config order.
5. **Be direct.** Romi's coaching preference is honest, not soft. See `about-me.md` → "How Claude Should Work With Me".
6. **Frame through Stoicism** when relevant. See `knowledge/stoicism.md` for reference.
7. **Respect time constraints.** Keep responses focused. Romi works full-time and has limited windows.
8. **When writing to files**, preserve existing content in other sections. Only modify the section you're updating.
9. **When analyzing trends**, always include specific numbers and dates. No vague assessments.
10. **When receiving unstructured input** (e.g., from SMS), match against category IDs in config first, then use AI classification as fallback.
